## Executive Summary

On first activation, the game lagged (stuttered) briefly when AEGIS kicked in. This caused a dt spike that disabled both the Kalman filter predict step AND the guidance controller simultaneously. With zero thrust commanded, the vehicle entered free-fall. The FDI, fed zero expected acceleration and observing gravity in the body-frame accelerometer data, misidentified the free-fall as multiple simultaneous engine failures and triggered HARD_ABORT via ISS-004.

## Root Cause

### Primary: Guidance gated on `skip_predict`

```python
# main.py:224
if self.state not in ["HARD_ABORT", "DEORBIT_BURN", "HYPERSONIC_COAST"] and not skip_predict:
    desired_wrench = self.guidance.compute_wrench(...)
else:
    desired_wrench = np.zeros(6)  # ← zeroed whenever skip_predict=True
```

When the game lagged on first tick, `dt = 0.208s >> 3 × 0.02 = 0.06s`. This set `skip_predict = True`. The guidance condition evaluated to `False`, zeroing `desired_wrench`. The control allocator received zero wrench → zero throttle → vehicle entered free-fall.

### Secondary: FDI misinterprets free-fall as engine failures

With zero throttle, `expected_accel = [0, 0, 0]`. The body-frame accelerometer read approximately `[−10, +10, −10] m/s²` — gravity acting on the falling vehicle. The deviation `≈ 17 m/s²` exceeded `FDI_THRESHOLD = 3.0`. After the persistence counter (50 ticks), FDI isolated engines 0, 1, 3, 4 as failed. `len(failed_indices) >= 2` triggered `HARD_ABORT` via ISS-004.

### Tertiary: Velocity never updated

The Kalman filter's predict step — which integrates acceleration into velocity via `B[3:6, 0:3] = dt * np.eye(3)` — was also gated on `skip_predict`. Since predict never ran, velocity remained at its initial value of `0.0` for the entire run, even after dt normalized. This locked the guidance system into trying to stop the vehicle from moving (which it already believed was stationary at velocity 0), while the vehicle was actually falling at velocity ≠ 0.

---

## Contributing Factors

### 1. dt spike on first tick
The game lagged on initial activation — a common scenario during loading screens or vessel stage separation. The 10× nominal dt (0.208s vs 0.02s) is well within plausible KSP physics lag.

### 2. Dual gating conflates two concerns
The `skip_predict` flag was used to gate both the Kalman filter predict step AND the guidance controller. These are independent decisions:
- **Predict step**: correctly skipped on large dt because the KF physics model diverges with non-physical dt values
- **Guidance controller**: should **never** be suppressed during lag — the vehicle still needs thrust to maintain control

### 3. No FDI disengagement during degraded state
FDI continued operating during dt spike events. With zero throttle commanded, `expected_accel` was identically zero, making any measured acceleration (including gravity during free-fall) appear as a massive fault.

## Resolution

**Immediate fix (committed as fixup):** Remove `and not skip_predict` from the guidance condition:

```python
# BEFORE (buggy):
if self.state not in ["HARD_ABORT", "DEORBIT_BURN", "HYPERSONIC_COAST"] and not skip_predict:

# AFTER (fix):
if self.state not in ["HARD_ABORT", "DEORBIT_BURN", "HYPERSONIC_COAST"]:
```

This ensures guidance runs continuously once in `POWERED_DESCENT`, regardless of dt spikes.

## Long-Term Recommendations

| Priority | Item | Description |
|----------|------|-------------|
| HIGH | Decouple guidance from `skip_predict` | Guidance should run in all flight phases regardless of dt. Consider a `degraded_mode` flag that commands hover (zero net force) during known degraded conditions rather than zeroing wrench entirely. |
| HIGH | Add FDI bypass during dt spikes | FDI's expected_accel will be stale/invalid during dt spikes. Consider skipping FDI fault detection (not isolation) when `skip_predict` is True, or hold the last known good `expected_accel` instead of computing from zero throttle. |
| MEDIUM | Add velocity fallback | If predict is skipped, consider falling back to numerical differentiation of position (altitude) for velocity estimation, or at minimum, do not trust the stale zero velocity estimate. |
| MEDIUM | Log skip_predict state | Add `skip_predict` to telemetry frame so the log clearly shows when predict and guidance are being skipped, making the root cause immediately visible. |
| LOW | Tune FDI persistence counter | 50 tick persistence (1 second at 50Hz) is conservative. Consider whether shorter persistence (e.g., 10-20 ticks) would reduce time-to-detection without false triggering. |

## Lessons Learned

1. **A single boolean gating multiple independent systems is a design smell.** `skip_predict` controlled KF predict, guidance, and indirectly FDI — three systems with different failure modes and different sensitivity to dt.

2. **Zero wrench is not a safe fallback.** Commanding zero thrust during a lag spike leaves the vehicle in free-fall. A safer degraded-mode command would be to maintain current altitude (hover) by computing the thrust required to exactly counter gravity.

3. **FDI requires a valid expected_accel to function.** When throttle is zero, expected_accel is zero, and any non-zero measured acceleration (gravity during free-fall) looks like a catastrophic fault. FDI is not designed to handle the "we're not thrusting at all" case.

4. **Telemetry shows zero velocity throughout, but this was never flagged as anomalous.** If the state estimate had been monitored for physical plausibility (e.g., altitude decreasing while velocity remains zero), the bug would have been visible immediately.
