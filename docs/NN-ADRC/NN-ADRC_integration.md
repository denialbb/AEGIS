# AEGIS Architecture Migration: NN-ADRC Integration Report

This report outlines the systems engineering process for migrating the existing AEGIS (Autonomous Estimation & Guidance Integrated System) architecture from a standard guidance solver to a Neural Network-assisted Active Disturbance Rejection Controller (NN-ADRC). It also details the required mathematical foundations and specific source literature needed to implement the control logic.

## Part 1: Integration Process Outline

The migration capitalizes on the existing decoupled Python/kRPC architecture, requiring module-specific upgrades rather than a complete rewrite.

### Phase 1: Upgrading the State Estimator
Currently, the AEGIS Discrete-Time Kalman Filter estimates a 6-DOF state vector containing only position and velocity:
\[
x = [x, y, z, v_x, v_y, v_z]^T
\]
To support the NN-ADRC, this module must be updated to also estimate and output clean acceleration data. Alongside the Kalman Filter, you must implement the Extended State Observer (ESO), which introduces an additional state variable exclusively dedicated to estimating total unknown disturbances in real-time.

### Phase 2: Implementing the NN-ADRC Core
The core guidance logic will be replaced with three interconnected ADRC components:
1. The **Transient Profile Generator (TG)** for smooth trajectory generation.
2. The **Weighted State Error Feedback (WSEF)** for reference acceleration calculation.
3. The **Computed Torque Model (CTM)**.

The Neural Network will sit parallel to this, taking the state estimator's position, velocity, and acceleration vectors as inputs to compute a compensatory acceleration vector (\(\Delta \ddot{\mathbf{r}}\)).

### Phase 3: Control Allocator Integration
The existing Control Allocator expects a 6-DOF Wrench vector:
\[
W = [F_x, F_y, F_z, \tau_x, \tau_y, \tau_z]^T
\]
and maps it to surviving engine throttles using a pseudo-inverse matrix (`numpy.linalg.pinv`).

To integrate the new controller, you will sum the baseline reference acceleration from the WSEF with the compensatory acceleration from the NN, convert this into the required Wrench vector format, and feed it directly into the existing Control Allocator.

### Phase 4: Adapting the FDI Module
The current Fault Detection & Isolation (FDI) module flags an engine as failed if the deviation between expected and measured acceleration exceeds a \(0.5\text{ m/s}^2\) threshold.

Because the NN-ADRC will instantly generate supplementary accelerations to suppress this deviation, the FDI will fail to detect the engine loss. The FDI logic must be rewritten to monitor the internal disturbance estimates generated by the ESO or the compensatory torque commanded by the Neural Network to flag failures.

### Phase 5: Neural Network Training
The network must be trained on the difference between the ideal Computed Torque Model and the actual failing plant. You will utilize your planned "Gremlin" background script to randomly force engine `thrustLimit` to 0 during automated KSP descents. This will generate the necessary training data, teaching the NN to output the exact compensatory acceleration (\(\Delta \ddot{\mathbf{r}}\)) needed for any given asymmetric thrust failure.

## Part 2: Required Mathematics and Sources

To implement the integration, refer to the following mathematical foundations and sources:

### 1. Active Disturbance Rejection & Neural Network Math
* **Source:** *NN Based Active Disturbance Rejection Controller for a Multi-Axis Gimbal System*
* **ESO Implementation:**
  You will need the state observer equations to estimate the system disturbances:
  \[
  \begin{aligned}
  \dot{z}_1 &= z_2 - \beta_{01} e \\
  \dot{z}_2 &= z_3 + b_0 u - \beta_{02} \text{fe} \\
  \dot{z}_3 &= -\beta_{03} \text{fe}
  \end{aligned}
  \]
  (where \(e = z_1 - y\) is the estimation error, and \(\beta_{01}, \beta_{02}, \beta_{03}\) are observer gains).
  *Note:* You must also implement the non-linear smoothing function \(\text{fal}(e, \alpha, \delta)\) to mitigate chattering.
* **NN Integration:**
  The mathematics for generating the compensatory acceleration vector:
  \[
  \Delta \ddot{\mathbf{r}} = [\Delta \ddot{\psi}_a, \Delta \ddot{\theta}_m]^T
  \]
  based on the position, velocity, and acceleration inputs is detailed in the NN training structure equations.

### 2. State Estimation & Filtering Math
* **Source:** *Extended Kalman Filtering* - Stanford University
* **Filter Matrices:**
  To upgrade your state estimator, you will need the exact matrix formulations for the prediction and update steps. Use the state transition model:
  \[
  x_{t \mid t-1} = A x_{t-1 \mid t-1}
  \]
  and covariance matrix prediction:
  \[
  P_{t \mid t-1} = A P_{t-1 \mid t-1} A^T + Q
  \]
  to process the noisy telemetry data.

### 3. Quaternion and Attitude Tracking Math
* **Sources:**
  * *Quaternion-Based Tracking Control Law Design for Tracking Mode*
  * *AN-1006 - Understanding Quaternions*
* **Attitude Error:**
  To avoid gimbal lock during aggressive abort maneuvers, attitude errors should be handled via quaternions. You will need the mathematical definition of the attitude error quaternion vector (\(\delta q\)), which represents the attitude difference between the current orientation (\(q\)) and the commanded orientation (\(q_c\)):
  \[
  \delta q = q_c^{-1} \otimes q
  \]
* **Euler Conversion:**
  For logging and display purposes, the equations to convert the resulting quaternions back into roll, pitch, and yaw Euler angles using the `atan2` and `arcsin` functions are required.

### 4. Baseline Architecture Interfaces
* **Sources:**
  * *AEGIS Core Architecture and Interface Specifications*

* **Wrench Matrix:**
  You must adhere to the 6-DOF state vector structure:

  x = [x, y, z, v_x, v_y, v_z]^T
